Rotas da API

  • /getUser - Essa rota é responsável por recuperar todas as informações do usuário, usando o método HTTP GET. Ela deve receber um parâmetro, chamado uid do usuário. A resposta será um json com as informações do usuário.
  • /user - Essa rota é responsável por gerenciar as operações relacionadas ao usuário, incluindo criar, atualizar e deletar usuários. Ela pode usar os métodos HTTP POST, PUT e DELETE, dependendo da operação a ser realizada. Os dados devem ser enviados em um json no padrão a seguir:
    { 'displayName': string, 'email': string, 'photoURL': string, 'sexo': string, 'uid': string}
    A resposta pode ser um json de sucesso com as informações ou erro com status 400 bad request ou 404 notfound.
  • /getTransaction - Essa rota é responsável por recuperar todas as transação do usuário, usando o método HTTP GET. Ela deve receber um parâmetro, chamado uid do usuário, para identificar qual transação deve ser recuperada. A resposta será um json com as informações da transação.
  • /cnpj - Essa rota é responsável por recuperar informações de um CNPJ específico usando o método HTTP GET. Ela deve receber um parâmetro, com um CNPJ válido, para identificar qual CNPJ deve ser recuperado. A resposta será um json com informações do CNPJ.
  • /transaction - Essa rota é responsável por criar novas transações usando o método HTTP POST. Ela deve receber informações em json, com o seguinte padrão:
    { "uid": string, "type": string, "type_especify": string, "date": string, "cash": string, "desc": string, "latitude": string, "longitude": string} .
    Como o valor da transação, a data e o usuário associado à transação. A resposta pode ser um json de sucesso ou erro.
  • /prediction - Essa rota é responsável por atualizar ou deletar uma transação existente. Ela deve usar os métodos GET, dependendo da operação a ser realizada. Ela deve receber o seguinte padrão em json:
    { "incoming":number, "age": number, "loan": number} .
    A resposta de sucesso é um objeto informando se o usuário foi aprovado ou negado, ou erro com status 400 ou 404.
  • /changeTransaction - Essa rota é responsável por atualizar ou deletar uma transação existente. Ela pode usar os métodos HTTP PUT ou DELETE, dependendo da operação a ser realizada. Ela deve receber o seguinte padrão em json:
    { "_id":string, "uid": string, "type": string, "type_especify": string, "date": string, "cash": string, "desc": string, "latitude": string, "longitude": string}.
    A resposta pode ser uma mensagem de sucesso ou erro com status 400 ou 404.

Casos de usos do projeto VERTICE

  • 1 - Cadastrar usuários

    Este caso de uso permite que os usuários criem uma conta no sistema, fornecendo e-mail e senha, ou se conectando via google.

  • 2 - Autenticação

    Este caso de uso permite que os usuários se autentiquem no sistema, geralmente fornecendo suas credenciais de login.

  • 3 - Gerenciamento de perfil

    Este caso de uso permite que os usuários logados no sistemas gerenciem suas informações de perfil, como alterar sua senha ou atualizar seus dados de perfil.

  • 4 - Gerenciamento de transações

    Este caso de uso permite que os usuários logados no sistema criem, editem e excluam suas transações no sistema.

  • 5 - Consulta de CNPJ

    Este caso de uso permite que os usuários busquem informações sobre empresas comunicando o cnpj.

  • 6 - Visualizar relatórios

    Este caso de uso permite que os usuários logados no sistema visualizem o dashboard indicando gráficos das transaçoes inseridas.

  • 7 - Calculadora de receita

    Este caso de uso permite que os usuários calculem a sua receita na abordagem 50/30/20.

  • 8 - Consultar Disponibilidade de empréstimo

    Este caso de uso permite que os usuários logados no sistema receba a informação que está apto para receber empréstimo bancários.

  • 9 - Mudança do tema do site

    Este caso de uso permite que os usuários configurem o site no modo noturno.

  • 10 - Selecionar latitude e longitude da transação no mapa.

    Este caso de uso permite que os usuários ao clicar em um mapa interativo faça que o sistema capiture as coordenadas e ao clicar em salvar é adicionada ao banco de dados.

Arquitetura do projeto

  • Vue.js é utilizado para construir a camada de interface do usuário do aplicativo. Ele fornece recursos avançados de renderização de componentes e gerenciamento de estado, tornando a interface do usuário reativa e fácil de gerenciar.
  • O front-end é implementado como uma aplicação estática e é implantado no GitHub Pages, permitindo uma fácil manutenção e escalabilidade.
  • Flask é usado como o framework de back-end, responsável por lidar com as requisições HTTP e fornecer as respostas apropriadas ao front-end. Ele também é responsável por gerenciar as operações de banco de dados.
  • MongoDB é utilizado como o banco de dados para armazenar e recuperar informações do usuário e dos dados do aplicativo. Ele é conectado ao Flask através de um driver, permitindo que o aplicativo execute operações CRUD (criar, ler, atualizar e deletar) nos dados.
  • Firebase Auth é utilizado para gerenciar a autenticação e autorização dos usuários. Ele permite que os usuários criem uma conta e façam login com suas credenciais, além de gerenciar a recuperação de senha e outras funcionalidades de segurança.
  • A instância EC2 na AWS é utilizada como a plataforma de hospedagem para o aplicativo Flask e o banco de dados MongoDB. Ela fornece a escalabilidade, segurança e gerenciamento de recursos necessários para o aplicativo.
  • Existe um servidor local utilizando o Flask contando com um modelo de aprendizado de máquina treinado para avaliar se um usário tem um empréstimo aprovado ou negado.

O projeto conta com uma arquitetura em quatro camadas:

Apresentação, negócios, persistencia e dados. Na camada de apresentação, o Vue.js é utilizado para construir a interface do usuário e publicado no GithubPages. Na camada de negócios, o Flask é utilizado para construir o back-end do aplicativo e instanciado na AWS EC2. Na camada de persistencia utilizamos um modelo de aprendizado de máquina treinado em um servidor local com Flask, para avaliar se um usuário tem um empréstimo aprovado ou negado. E na camada de dados, o MongoDB é utilizado como banco de dados e o Firebase Auth como sistema de autenticação. Essa arquitetura permite uma boa separação entre as camadas, facilitando a manutenção e escalabilidade do aplicativo. A camada de apresentação é responsável por exibir a informação para o usuário e obter suas ações, a camada de negócios é responsável por processar as ações do usuário e obter ou armazenar informações no banco de dados e a camada de dados é responsável por armazenar e recuperar informações persistentes.
